# Internacionalització (i18n) amb next-intl v4

## ⚠️ Lliçons apreses (Postmortem — 02/06/2026)

### El problema que vam tenir
Després d'implementar `next-intl` v4 amb `[locale]` routing, el `LanguageSwitcher` canviava correctament la URL entre `/ca`, `/es`, `/en`, `/fr`... **però tots els textos de la interfície es continuaven renderitzant sempre en català**, independentment de l'idioma de la URL.

### Per què passava?
Next-intl v4 introdueix canvis importants respecte a v3 que, si no es respecten, fan que el sistema d'i18n **sembli funcionar** (el middleware canvia la URL correctament) però **no funcioni realment** (els components sempre reben els missatges del `defaultLocale`).

Les 6 causes arrel del problema van ser:

| # | Fitxer | Error | Consequència |
|---|--------|-------|--------------|
| 1 | `i18n/request.ts` | Usava `locale` directament (API de v3) en lloc de `requestLocale` + `hasLocale()` | El sistema de missatges no es resolia correctament |
| 2 | `app/[locale]/layout.tsx` | NO cridava `setRequestLocale(locale)` | `getMessages()` sempre retornava els missatges de `ca` |
| 3 | Tots els `page.tsx` dins de `[locale]/` | NO cridaven `setRequestLocale(locale)` | `useTranslations` als Server Components sempre rebia `ca` |
| 4 | `proxy.ts` | Usava `export function proxy()` en lloc de `export default` | El middleware no s'integrava correctament amb next-intl |
| 5 | `next.config.ts` | `createNextIntlPlugin()` sense path | El plugin no trobava la configuració |
| 6 | `app/[locale]/layout.tsx` | No validava el locale amb `hasLocale()` | Rutes invàlides passaven sense error |

### Per què era difícil de detectar?
- El **build passava** correctament (40 pàgines generades).
- Els **tests passaven** (perquè mockegen `next-intl`).
- L'**ESLint no deia res**.
- La **URL canviava** correctament.
- L'únic símptoma era que **els textos no canviaven**, cosa que semblava un problema de traduccions i no de configuració.

## ✅ Setup correcte d'i18n amb next-intl v4 (App Router)

### 1. `i18n/routing.ts` — Definició central de locales

```ts
import { defineRouting } from 'next-intl/routing';

export const routing = defineRouting({
  locales: ['ca', 'es', 'en', 'fr'] as const,
  defaultLocale: 'ca',
  localePrefix: 'always', // Tots els locales porten prefix (/ca, /es, /en, /fr)
});

export type Locale = (typeof routing.locales)[number];
```

### 2. `i18n/request.ts` — Càrrega de missatges al servidor

⚠️ **IMPORTANT:** Usar `requestLocale` (no `locale`) i validar amb `hasLocale()`.

```ts
import { getRequestConfig } from 'next-intl/server';
import { hasLocale } from 'next-intl';
import { routing } from './routing';

export default getRequestConfig(async ({ requestLocale }) => {
  const requested = await requestLocale;
  const locale = hasLocale(routing.locales, requested)
    ? requested
    : routing.defaultLocale;

  const messages = (await import(`@/locales/${locale}.json`)).default;

  return {
    locale,
    messages,
  };
});
```

### 3. `proxy.ts` — Middleware (Next.js 16+)

⚠️ **IMPORTANT:** Usar `export default` (no una funció anomenada `proxy`).

```ts
import createMiddleware from 'next-intl/middleware';
import { routing } from './i18n/routing';

export default createMiddleware(routing);

export const config = {
  // Exclou API, _next, fitxers estàtics, assets i media (audio/video)
  matcher: '/((?!api|_next|_vercel|static|assets|favicon.ico|robots.txt|.*\\.(?:webp|png|jpg|jpeg|gif|svg|ico|css|js|woff|woff2|ttf|eot|pdf|zip|mp4|webm|mov|mp3|wav|ogg)).*)',
};
```

### 4. `next.config.ts` — Plugin de next-intl

⚠️ **IMPORTANT:** Passar el path al `getRequestConfig` explícitament.

```ts
import type { NextConfig } from "next";
import createNextIntlPlugin from 'next-intl/plugin';

const nextConfig: NextConfig = {
  images: {
    unoptimized: true,
    remotePatterns: [],
  },
};

const withNextIntl = createNextIntlPlugin('./i18n/request.ts');
export default withNextIntl(nextConfig);
```

### 5. `app/[locale]/layout.tsx` — Layout amb Provider

⚠️ **CRÍTIC:** Cridar `setRequestLocale(locale)` **ABANS** de `getMessages()`. Validar el locale amb `hasLocale()`.

```tsx
import React from 'react';
import { NextIntlClientProvider, hasLocale } from 'next-intl';
import { getMessages, setRequestLocale } from 'next-intl/server';
import { notFound } from 'next/navigation';
import { routing } from '@/i18n/routing';

export function generateStaticParams() {
  return routing.locales.map((locale) => ({ locale }));
}

export default async function LocaleLayout({
  children,
  params,
}: {
  children: React.ReactNode;
  params: Promise<{ locale: string }>;
}) {
  const { locale } = await params;

  if (!hasLocale(routing.locales, locale)) {
    notFound();
  }

  // ⚠️ AQUESTA CRIDA ÉS OBLIGATÒRIA
  setRequestLocale(locale);

  const messages = await getMessages();

  return (
    <NextIntlClientProvider messages={messages} locale={locale}>
      {children}
    </NextIntlClientProvider>
  );
}
```

### 6. Tots els `page.tsx` dins de `[locale]/`

⚠️ **CRÍTIC:** Cada `page.tsx` ha de cridar `setRequestLocale(locale)` quan usa `useTranslations` o altres APIs de next-intl.

```tsx
import { use } from 'react';
import { setRequestLocale } from 'next-intl/server';
import { useTranslations } from 'next-intl';

export default function MyPage({ params }: { params: Promise<{ locale: string }> }) {
  const { locale } = use(params);

  // ⚠️ AQUESTA CRIDA ÉS OBLIGATÒRIA
  setRequestLocale(locale);

  const t = useTranslations('myNamespace');

  return <h1>{t('title')}</h1>;
}
```

Per a pàgines dinàmiques com `[slug]/page.tsx`, generar els params estàtics per **tots els locales**:

```tsx
export function generateStaticParams() {
  return SERVICES_DATA.flatMap((service) =>
    routing.locales.map((locale) => ({
      locale,
      slug: service.slug,
    }))
  );
}
```

### 7. `components/ui/LocaleLink.tsx` — Link amb prefix automàtic

⚠️ **IMPORTANT:** Utilitzar `LocaleLink` en lloc de `next/link` per a tots els enllaços interns. `LocaleLink` afegeix automàticament el prefix de locale a la URL.

```tsx
import LocaleLink from '@/components/ui/LocaleLink';

// ✅ Correcte — LocaleLink afegeix /ca/, /es/, /en/, /fr/ automàticament
<LocaleLink href="/cataleg">Catàleg</LocaleLink>
<LocaleLink href="/contacte">Contacte</LocaleLink>

// ❌ Incorrecte — Mai fer prefix manual de locale
<Link href="/ca/cataleg">Catàleg</Link>
<Link href="/es/contacte">Contacte</Link>
```

**Quan utilitzar `next/link` en lloc de `LocaleLink`:**
- Només per enllaços externs (a altres dominis).
- Per enllaços a `#anchor` dins de la mateixa pàgina.

## 🧪 Testing

### Mock global a `jest.setup.ts`

```ts
jest.mock('next-intl', () => ({
  useTranslations: () => (key: string) => key,
  useLocale: () => 'ca',
  useMessages: () => ({}),
  NextIntlClientProvider: ({ children }: { children: React.ReactNode }) => children,
  hasLocale: (_locales: string[], requested: string) => requested === 'ca',
}));
```

> ℹ️ El mock retorna la **clau** com a valor (`t('foo.bar')` → `'foo.bar'`). Per tant, els tests han de buscar les claus de traducció, no els textos finals.

### Plantilla de test per a un component amb traduccions

```tsx
import { render, screen } from '@testing-library/react';
import MyComponent from '@/components/MyComponent';

describe('MyComponent', () => {
  it('renderitza el títol', () => {
    render(<MyComponent />);
    // t('myNamespace.title') → retorna 'title'
    expect(screen.getByRole('heading', { name: /title/i })).toBeInTheDocument();
  });
});
```

## 📋 Checklist obligatori per a cada nova pàgina/component

Quan creïs una nova pàgina sota `[locale]/` o un component que usa traduccions:

- [ ] La pàgina rep `params: Promise<{ locale: string }>`
- [ ] Fa `const { locale } = use(params)` (client) o `await params` (server)
- [ ] Crida `setRequestLocale(locale)` **abans** de qualsevol `useTranslations` o `getMessages`
- [ ] Si és una pàgina dinàmica (`[slug]`), `generateStaticParams` itera per **tots** els `routing.locales`
- [ ] Els fitxers JSON a `locales/{ca,es,en,fr}.json` contenen totes les claus usades
- [ ] Els tests busquen les **claus de traducció** (`/title/i`), no els textos finals
